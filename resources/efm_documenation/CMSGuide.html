APIs for Case Management Systems

This guide is targeted primarily towards CMS vendors who are building an integration to the EFM via web services. However, vendors implementing the one-way SFTP integration will find the RecordDocketing API documentation useful since the request XML is identical with the exception of the document binaries. Please refer to Appendix A of the OFS CMS Specification.pdf for more details about SFTP and document binaries.

IMPORTANT NOTES

  • The schema folder consists of the ECF 5 WSDLs which define the messages, bindings, and ports. This folder also contains several Tyler specific WSDLs which define additional operations that are not provided by the ECF specification.
  • The service folder contains Tyler implementation specific WSDLs which imports the WSDLs from the schema folder and Tyler extensions where appropriate. These WSDLs define the service and security policy.
  • The WSDLs without the Tyler prefix are our implementation of ECF 5. That is, they only contain bona-fide ECF 5 operations but might include extension elements. Key point: no non-ECF operations are in these WSDLs. The WSDLs with the Tyler prefix are new operations that Tyler has added to cover functionality needs that ECF does not address.
  • The EFM implementation of ECF does not support the xsi:type attribute in XML; instead, the ECF message should explicitly define the common ECF namespace prefix for the element or define it on the element. Use of the xsi:type attribute can cause parsing to fail, which could result in unknown error messages.
  • CodeIDs used in a test or stage environment will be different than the corresponding code's CodeIDs in production.
  • URLs included herein and in the sample files are for demonstration purposes only. The actual URLs vary by market.

          The EFM makes calls to the CMS in order to query for information pertaining to existing cases. Regarding secured cases, the CMS should adhere to jurisdictional requirements as to whether to return such cases and masking sensitive data.

            • GetCase

            GetCaseList

            GetCaseList queries for a list of cases within a particular court's CMS, matching on either the case number or a case participant's name. A search by case participant name is a search for an individual or an organization, not an individual within an organization. If a CMS receives a new case from the EFM, that case should not be included in the GetCaseList results until the CMS has completed the callbacks for the initial filing.

            The intent is to retrieve a subset of cases for the specified criteria, not a complete historical record of all cases. Therefore, the CMS must impose an upper boundary of 1000 results. The CMS must return the most recent cases in descending date filed (reverse chronological) order.

            EFM Behaviors

            • For court locations that do not provide the EFM with the ability to search for cases, the EFM will search its own records. If no results are found, the EFM will return error code 136. This indicates that the filer may still file into cases not found (as a filing into a non-indexed case)
            • For court locations that do provide the EFM with the ability to search for cases, the EFM will call upon the CMS to do so. If the call to the CMS fails, the EFM will search its own records to provide results, and will return error -11 to indicate that the CMS was unavailable, therefore imposing the caveat that the information returned may be outdated.

              RecordDocketing

              RecordDocketing submits one or more filings to the court CMS for docketing following acceptance by the clerk from within the EFM. This section includes elements of the message that apply regardless of case type. Tyler Implementation staff will configure each case category to use a particular ECF case type schema which will be used for both initial filings and subsequent filings.

              The EFM uses some elements that are specific to the ECF case type schemas, but only for initiating the following case types:

              • Appellate cases
              • Criminal cases
              • Citation cases

              NOTE: The EFM does not currently support filing of new Citation cases. Citations are only supported in conjunction with a criminal case filing.

              The EFM does not use elements specific to the ECF case type schemas for subsequent filings, therefore subsequent filings are functionally equivalent regardless of case type.

                        SecureCase

                        Under normal operation a web-service enabled integration to the CMS allows the CMS to control access to cases by simply not returning cases that are considered secure. However, in the event that the integration to the CMS is unavailable, search results will be provided from the EFM's records.

                        This API enables the CMS to notify the EFM as to whether or not a case should be considered secure. Secured cases will not be included in the EFM's search results, thereby making it unavailable for e-filing when the CMS is unavailable.
